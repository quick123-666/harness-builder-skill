## 二、三层防护模型

### Layer 1: 系统配置层

**OpenClaw elevated 权限配置：**
```json
{
  "tools": {
    "elevated": {
      "enabled": true,
      "allowFrom": {
        "webchat": []
      }
    }
  }
}
```

```json
{
  "protectedFiles": [
    ".env",
    ".env.*",
    "*.key",
    "*.pem"
  ],
  "blockedCommands": [
    "sudo *",
    "rm -rf *",
    "git push --force *"
  ]
}
```

## 三、PWR 循环架构

```
┌─────────────────┐
│  Plan Agent     │
│  (只读, opus)   │
│  - 制定计划      │
│  - 风险评估      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Work Agent     │
│  (隔离, sonnet) │
│  - 执行计划      │
│  - 记录日志      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Review Agent   │
│  (只读, opus)   │
│  - 审查结果      │
│  - 质量验证      │
└─────────────────┘
```

**Plan Agent 配置：**
- 权限：只读（Read / Grep / Glob）
- 模型：opus
- 职责：制定详细计划

**Work Agent 配置：**
- 权限：读写（Read / Write / Edit / Bash）
- 模型：sonnet
- 职责：隔离执行计划

**Review Agent 配置：**
- 权限：只读（Read / Grep / Glob / Bash for tests）
- 模型：opus
- 职责：审查执行结果

## 四、防护规则体系

### 配置层（deny）

| 规则 | 触发条件 | 行为 | 原因 |
|------|---------|------|------|
| R01 | `sudo *` | deny | 防止权限提升 |
| R02 | 敏感文件 | deny | 防止信息泄露 |
| R06 | `git push --force` | deny | 防止代码丢失 |

### 行为层（deny）

| 规则 | 触发条件 | 行为 | 原因 |
|------|---------|------|------|
| R03 | 删除核心文件 | deny | 防止核心代码丢失 |
| R10 | 非白名单网络 | deny | 防止数据外泄 |
| R11 | 非签名脚本 | deny | 防止恶意代码 |

### 行为层（ask）

| 规则 | 触发条件 | 行为 | 原因 |
|------|---------|------|------|
| R04 | 修改核心配置 | ask | 确保用户知情 |
| R05 | 删除操作 | ask | 防止误删 |

### 行为层（warn）

| 规则 | 触发条件 | 行为 | 原因 |
|------|---------|------|------|
| R12 | 大文件写入 | warn | 防止资源耗尽 |
| R13 | 频繁操作 | warn | 防止无限循环 |

## 八、风险与缓解

### 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| elevated 权限配置失败 | 高 | 中 | 提供详细配置指南 |
| 审计日志性能问题 | 中 | 低 | 异步写入、批量处理 |
| PWR 循环隔离失败 | 高 | 低 | 严格测试 sessions_spawn |

### 运营风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 用户不理解三层防护 | 中 | 高 | 提供详细文档和示例 |
| 审计日志过大 | 低 | 中 | 定期归档、压缩存储 |
